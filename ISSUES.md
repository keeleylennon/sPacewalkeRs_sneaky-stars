### Issue G1 — `enhancement`

**Title:** Analysis and visualisation logic are mixed in a single script — consider separating into two files

**Description:**

Both `src/analyse_spacewalks.py` and `src/analyse_spacewalks.R` combine two distinct responsibilities in a single file: data loading and processing (parsing dates, converting durations, splitting crew names, filtering) and figure generation (defining colours, building plots, saving outputs). As scripts grow, this makes them harder to read, test, and reuse — if a collaborator wants to use the processed data for a new figure, they have to run or import the entire script.

- `src/process_spacewalks.py` / `src/process_spacewalks.R` — loads the raw data and returns a clean, analysis-ready dataframe
- `src/visualise_spacewalks.py` / `src/visualise_spacewalks.R` — imports the processed data and generates all figures

This mirrors standard practice in data science workflows and makes each script easier to understand, test, and build upon independently.

**Note:** This is an open-ended refactor — there is no single correct solution. Consider which steps belong to processing versus visualisation, and how the two scripts will share data (e.g. saving an intermediate `.csv`, or importing functions directly).

### Issue G2 — `enhancement`

**Title:** Raw data is committed to the repository — should be archived separately and downloaded at runtime

**Description:**

The file `data/data.json` is committed directly to this repository. This is a common pattern in research code, but it conflates two things that should be kept separate: the **code** (which belongs in a Git repository) and the **data** (which belongs in a dedicated data archive with its own DOI).

Committing raw data to a repository creates several problems:
- It bloats the repository history — every version of the data is stored forever in Git, even if the file never changes
- Data has no independent citation or persistent identifier
- If the data ever needs to be updated or versioned separately from the code, there is no clean way to do so
- For larger datasets this approach does not scale at all

The standard practice is to archive the data in a repository such as Zenodo or Figshare, then download it at runtime from a stable URL. This separates concerns cleanly: the code repository is citable via its own DOI, and the data repository is citable via its own DOI — consistent with the FAIR principles covered in Session 4.

**Tasks:**
- [ ] Deposit `data/data.json` in a data repository (e.g. Zenodo, Figshare) and obtain a permanent download URL
- [ ] Update both `src/analyse_spacewalks.py` and `src/analyse_spacewalks.R` to download the data file from that URL at runtime rather than reading from a local path
- [ ] Add `data/` to `.gitignore` so it is no longer tracked
- [ ] Remove the committed `data/` directory from the repository history
- [ ] Update `README.md` to link to the data record rather than describing a local file
